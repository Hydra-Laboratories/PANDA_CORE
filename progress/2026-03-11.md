## Import Fix: Eliminate all `from src.` imports

### Problem
Zoo's venv couldn't resolve `src` as a module (`ModuleNotFoundError: No module named 'src'`). PANDA_CORE adds `src/` to sys.path (via pyproject.toml `pythonpath = ["src"]`), so all imports must use direct package names (e.g., `from board.board import Board`), never `from src.board.board import Board`.

Previous sessions fixed source files under `src/` but left test files and setup scripts with `from src.` imports. This caused `isinstance` mismatches — Python treated `src.deck.labware.vial.Vial` and `deck.labware.vial.Vial` as different classes, causing silent failures in bounds validation.

### Files Fixed (8 total)
**Test files (6):**
- `tests/validation/test_bounds_validation.py` — 8 import lines
- `tests/setup/test_integration.py` — 5 import lines + 3 `import src.` reload lines
- `tests/setup/test_protocol_setup.py` — 9 import lines + 3 `import src.` reload lines
- `tests/gantry/test_gantry_config.py` — 1 import line
- `tests/gantry/test_loader.py` — 3 import lines
- `tests/gantry/test_yaml_schema.py` — 1 import line

**Setup scripts (2):**
- `setup/run_protocol.py` — 3 import lines
- `setup/validate_setup.py` — 9 import lines

### Result
- **0 occurrences** of `from src.` or `import src.` remain in the codebase
- **571 tests pass**, 0 failures
- Zoo venv should now resolve all imports correctly

### Key Convention
All imports within PANDA_CORE must use package names directly:
- `from board.board import Board` (correct)
- `from src.board.board import Board` (wrong — will break in zoo venv)

## Gantry: WPos/MPos Coordinate Convention & GRBL Settings Validation

### Problem
Zoo UI was reporting negative position numbers. The gantry uses GRBL's $3=2 (Y-axis direction inverted), which makes MPos negative for Y. Multiple approaches were tried before settling on the correct design.

### Final Design
1. **On connect**: Validate GRBL settings against YAML, switch to WPos reporting (`$10=0`)
2. **On home**: Run homing strategy, then set WPos=(x_max, y_max) from working_volume config
3. **Convention**: Home position = (max, max). As gantry moves into workspace, WPos decreases toward (0,0). All positions are positive.
4. **G54 offset persists in GRBL EEPROM** across power cycles — no re-homing needed on reconnect unless machine was power-cycled.

### Files Modified
- **`src/gantry/gantry.py`** — Added `get_position_info()`, `_extract_status()`, `jog()`, `home_xy()`, `_switch_to_wpos_reporting()`, `_set_home_wpos()`, `_validate_grbl_settings()`, `_query_raw_status()`. Added module-level regex patterns for GRBL status parsing.
- **`src/gantry/yaml_schema.py`** — Added `GrblSettingsYaml` model (all GRBL settings as optional fields) and `GRBL_FIELD_TO_SETTING` mapping dict. Added `grbl_settings` field to `GantryYamlSchema`.
- **`src/gantry/gantry_config.py`** — Added `expected_grbl_settings` field to `GantryConfig`.
- **`src/gantry/loader.py`** — Updated to pass GRBL settings from YAML → GantryConfig.
- **`src/gantry/gantry_driver/driver.py`** — Modified `home_xy_hard_limits()` to set WPos=0 at home (neutral), letting `Gantry._set_home_wpos()` handle the correct offset.
- **`src/protocol_engine/setup.py`** — Added `run_protocol()` wrapper function (zoo depends on it).
- **`configs/gantry/genmitsu_3018_PRO_Desktop.yaml`** — Added `grbl_settings` section.
- **`configs/gantry/genmitsu_3018_PROver_v2.yaml`** — Added `grbl_settings` section.
- **`pyproject.toml`** — Added `validation` and `validation.*` to packages include list.

### Tests Added
- **`tests/gantry/test_gantry.py`** — `TestGrblSettingsValidation` (6 tests): passes when settings match, raises on critical mismatch (dir_invert_mask, max_travel), skips when no settings, warns on non-critical mismatch, ignores unspecified settings.
- **`tests/gantry/test_yaml_schema.py`** — `TestGrblSettingsYaml` (5 tests): optional by default, parses correctly, all fields optional, rejects extra fields, handles boolean fields.

### GRBL Settings Validation
On connect, `_validate_grbl_settings()` compares YAML `grbl_settings` against live controller values. Critical mismatches (direction inversion, max travel, steps/mm) raise `MillConnectionError`. Non-critical mismatches log errors but don't block connection.
