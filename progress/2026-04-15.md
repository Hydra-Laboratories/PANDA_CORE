# 2026-04-15 — ASMI dual indentation PR review fixes

## Context

Branch: `feat/asmi-dual-indentation-protocol`. This feature adds a
`measure_with_return` flag to `ASMI.indentation` so that after descending to
`z_limit`, the instrument also records upward samples back to
`measurement_height`, tagging each sample with `direction="down"` or `"up"`.
Goal is to align CubOS with the dual-indentation functionality on the
asmi-new main branch.

Ran `/pr-review-toolkit:review-pr` first. Four agents (code-reviewer,
pr-test-analyzer, silent-failure-hunter, comment-analyzer) flagged a mix of
critical loop-safety issues, schema inconsistencies, and test gaps.

## Work done

All fixes below implemented against the unstaged working-tree changes.

### Driver (`src/instruments/asmi/driver.py`)

- Added `_step_count_bound(z_upper, z_lower, step_size)` helper at module
  scope. Returns an upper bound on the number of steps required to cross a
  Z span, with a safety margin. Raises `ValueError` if `step_size <= 0`.
- Replaced the online descent `while True` with a `for _ in range(max_steps)`
  bounded by `_step_count_bound`. Added `else:` clause logging a warning if
  the cap is hit before reaching `z_target`.
- Replaced the online return `while True` with the same bounded loop. Added
  an in-loop guard: if `gantry.get_coordinates()` shows Z did not advance
  after a `move_to`, log a warning and break (stall detection). This is
  what prevented the original code from terminating when the gantry refused
  to move.
- Made `direction` unconditional on every sample in both online and offline
  paths. Previously it was only emitted when `measure_with_return=True`,
  producing heterogeneous dicts.
- Added an info log when the return sweep starts (observability request
  from review).
- Rewrote offline descent/return as integer-indexed `for` loops (`range(1,
  n+1)` with `max(..., z_limit)` / `min(..., measurement_height)` clamps).
  Eliminates the float-accumulation drift the review flagged.
- Tightened the `indentation` docstring to describe the return target,
  every sample being tagged, and the full return-value schema.

Per the user's guidance ("return force is less than downward force") we did
**not** add a force-limit check to the return stroke. Descent force check
is unchanged.

### Normalizer (`src/protocol_engine/measurements.py`)

- Added a warning log when `direction` tags are present on only *some*
  steps. This catches mid-run corruption where part of a return sweep was
  lost. Legacy fully-untagged payloads remain silent; the default to
  `"down"` for missing entries is preserved.

### Tests

- `tests/instruments/test_asmi.py`:
  - Migrated the existing indentation tests to pass overrides via kwargs
    (`z_limit=`, `measurement_height=`, `step_size=`) instead of mutating
    private attributes.
  - `test_indentation_offline_emits_direction_unconditionally`: every
    sample carries `direction="down"` even without return mode.
  - `test_indentation_offline_return_preserves_ordering_and_monotonicity`:
    all `down` samples precede all `up` samples; descent Z is strictly
    decreasing, return Z strictly increasing, return terminates exactly at
    `measurement_height`.
  - `test_indentation_offline_return_no_float_drift`: step_size that does
    not evenly divide the range still terminates cleanly at
    `measurement_height`.
  - New `TestASMIOnlineIndentation` class covers the `_offline=False` path
    for the first time, using a minimal `_FakeOnlineGantry` stub plus
    `mock.patch.object` on `get_force_reading` / `get_baseline_force`.
    Two tests: happy path with both directions recorded, and a
    stall-simulating gantry that would previously have spun forever.
- `tests/protocol_engine/test_measurements.py`:
  - `test_normalize_asmi_partial_direction_defaults_missing_to_down`:
    mixed-tag step lists coerce missing entries to `"down"`.

### Docs

- `docs/board.md`: changed `measure_with_return=true` (YAML-style) to
  `True` in the Python-signature table entry and expanded the description
  to note that every measurement carries a `direction` tag.
- `docs/protocol.md`: unchanged — the YAML example already uses lowercase
  `false` correctly.

## Issues found and how resolved

- **Stall test hung on first run.** Confirms the infinite-loop bug the
  review flagged. Added `pytest-timeout` (`--timeout=10`) to the test
  environment to catch future regressions fast, and the fix (iteration cap
  plus stall-break in the return loop) resolves it cleanly.
- **Python 3.9 venv checked in at `venv/` cannot run the codebase** (pipe
  syntax `float | None` requires 3.10+). Built a fresh 3.11 venv at
  `/tmp/cubos-venv-311` for test execution only; no repo changes.

## Test results

- `pytest tests/instruments/test_asmi.py tests/protocol_engine/test_measurements.py --timeout=10` →  24 passed.
- Full suite `pytest --timeout=30` → **875 passed, 15 skipped**, no failures.

## Follow-ups

- None required for this PR. The descent iteration cap is currently only a
  warning; if a future review wants it to raise, the `else:` branches are
  the single place to change.

# 2026-04-15 — Labware holder fixes (typed contained labware)

## Context

Branch: `labware-holder-fixes`. `VialHolder` and `WellPlateHolder` didn't
carry typed references to the vials/plates they hold, making it awkward to
compute the true top-Z of a contained vial or plate (e.g. for approach
heights, collision checks). The existing YAML schema embedded vials / plates
as nested dicts under the holder, so each vial had no stable global identity.

The design, agreed before coding:

- Vials and well plates become **top-level labware** in `deck.yaml`, each
  with explicit `z`. `vial.location.z` is the **bottom-center** Z (seat
  surface of the vial).
- Holders reference contained labware **by name**
  (`vials: [name, ...]`, `well_plate: name`).
- `VialHolder.vials` is `Dict[str, Vial]`; `WellPlateHolder.well_plate` is
  `Optional[WellPlate]`.
- Convenience helpers: `VialHolder.get_vial_top_z(name)` and
  `WellPlateHolder.get_plate_top_z()`; `Vial.get_bottom_center()` and
  `Vial.get_top_center()`.
- Back-reference: `Vial.holder` / `WellPlate.holder` (non-serialized,
  populated by the loader).
- Loader validates at load time that each contained vial's `location.z`
  matches `holder.location.z + labware_seat_height_from_bottom_mm` — hard
  error on drift, wrong-type reference, unknown reference, or duplicate
  ownership.

  the holder-construction kwargs in `_build_holder`; the loader's pass-2

# 2026-04-15 — PR review fixes for labware-holder-fixes

Applied the full set of findings from `/pr-review-toolkit:review-pr`.

## Design-level changes

- **Moved z-drift validation from the loader into the holder types.**
  `VialHolder` and `WellPlateHolder` now carry `@model_validator(mode="after")`
  that enforces `vial.location.z == holder.location.z +
  labware_seat_height_from_bottom_mm` (±1 µm), sets the `vial.holder` /
  `well_plate.holder` back-reference, and re-derives `contained_labware`
  from the typed field. Holders with named children must set
  `labware_seat_height_from_bottom_mm` (hard error if missing instead of
  silently skipping validation). Loader-side drift / back-ref / mirror
  logic removed; the loader now only handles existence, type, and
  cross-holder uniqueness (which genuinely need the global inventory).
- **`validate_assignment=True`** on both holder subclass `model_config`s —
  post-construction mutation of `vials` / `well_plate` re-runs the
  validators, so the loader's `holder.vials = resolved` assignment is
  equivalent to running the validator. A single base-class validator that
  writes `self.geometry` inside a model_validator was switched to
  `object.__setattr__` to avoid infinite recursion.
- **Z-drift tolerance constant** promoted to `_SEAT_Z_TOLERANCE_MM` in
  `holder.py` (1 µm, with a comment explaining why) and imported into
  both holder subclass validators.

## Loader hardening

- `load_deck_from_yaml_safe` now re-raises `DeckLoaderError` unchanged
  (previously double-wrapped) and only wraps `(ValidationError,
  yaml.YAMLError, FileNotFoundError)`. Unexpected `OSError`/programming
  bugs propagate with their native traceback.
- `_resolve_load_names` no longer catches `ValueError` — "unknown
  load_name" is now detected by explicit membership check against
  `get_supported_definitions()`, so a malformed definition config bubbles
  up with its real message instead of being mis-attributed as "Unknown
  load_name".
- `_build_holder`'s `kwargs.pop` calls are now narrowed per holder type
  (`VialHolderYamlEntry` → pop `vials`, `WellPlateHolderYamlEntry` → pop
  `well_plate`) with a comment explaining the YAML/Python field-name
  collision.
- Pass-2 `assert isinstance` checks replaced with explicit
  `raise DeckLoaderError("Internal loader error: …")` so the guards still
  fire under `python -O`.
- The "unsupported entry type" fallback `raise TypeError` became a
  `DeckLoaderError` with "please file a bug" guidance.
- Loader now validates that a held `WellPlate` has `height_mm` set —
  top-Z math would otherwise fail silently at runtime.

## Type + API polish

- `WellPlate.height_mm` description corrected to "Overall plate thickness
  in millimeters (bottom surface to top surface)" — the previous docstring
  claimed it was an absolute Z position. Added `height_mm` to the positive-
  value field validator so 0/negative values error at the field level
  rather than deep in `BoundingBoxGeometry`.
- `get_vial_top_z` / `get_plate_top_z` docstrings now list the exceptions
  they raise. `get_vial_top_z` error message lists the known vial names.
- `VialHolder` class docstring relaxed from "linear array" to

## panda_deck.yaml

- Dropped the per-vial/per-plate arithmetic comments that hardcoded

  pointing at the loader for enforcement.
- `tip_rack_a` / `tip_rack_b` calibration comments tightened.

## AGENTS.md

  `tip_holder:` YAML type that have never existed on this branch.

## Tests

# 2026-04-15 — Round-2 PR review fixes for labware-holder-fixes

Second pass of `/pr-review-toolkit:review-pr` flagged a set of issues that
were introduced by the Round-1 refactor itself (validate_assignment
interactions, `object.__setattr__` escape hatches for a stored
`contained_labware` field, stale back-references, loader `except ValueError`
mis-attributing all failures to "z-drift", etc.). All fixes below land in the
`cubos-labware` worktree (Charles's parallel-agent worktree for this work).

## Architectural

- **`HolderLabware.contained_labware` is now a derived property**, not a
  stored field. Subclasses override `_iter_contained_labware()` to contribute
  their children; the base `contained_labware` property wraps it and returns
  a fresh dict each call. This eliminates three `object.__setattr__` escape
  hatches (`vial_holder`, `well_plate_holder` × 2), the redundant
  `_validate_contained_labware_names` validator, and the footgun where a
  caller's `contained_labware=...` was silently discarded by the subclass
  validator.
- **Back-reference lifecycle is now managed.** Each holder tracks its
  previously-held labware in a `PrivateAttr` and, when `vials` / `well_plate`
  is reassigned (via `validate_assignment=True`), clears `.holder` on any
  labware that's no longer held. Prevents dangling back-references after
  `holder.vials = new_dict`.
- **Cross-holder single-ownership is now a type invariant**, not just a
  loader concern. Each holder's validator rejects labware whose `.holder` is
  set to a different holder, with a message like `"vial 'v1' is already held
  by another VialHolder ('first'); each vial may belong to at most one
  holder."` The loader's `claimed` dict still runs first (it gives cleaner
  cross-key errors), but programmatic callers now get the same guarantee.
- **`WellPlateHolder` now requires `well_plate.height_mm` at construction**
  when a plate is assigned. The loader-side check stays for YAML-oriented
  "How to fix" hints; the type-level check is defense-in-depth.
- **`Vial.holder` / `WellPlate.holder` back-references are now validated in
  sequence**: the validators do a full validate-all pass before any mutation,
  so a drift failure at vial #5 no longer leaves vials #1-4 with stale
  back-references pointing at a rejected holder.

## Loader hardening (round 2)

- **`except ValueError` no longer hardcodes "align z" advice.** The holder
  validators' own messages already name the specific invariant (drift,
  slot_count, name mismatch). The loader just prepends its context:
  `"❌ vial_holder 'h': <validator message>"`. No more misleading
  "align z" hint for a slot_count overflow.
- **`_resolve_load_names` caches `get_supported_definitions()`** outside
  the per-entry loop so a large deck no longer re-reads `registry.yaml`
  from disk once per `load_name` entry.
- **`assert isinstance` defensive re-checks in pass 2 deleted**. Pass 1
  built each holder with `model_class=VialHolder` / `WellPlateHolder`, so
  the runtime type is guaranteed; the re-check was dead code.
- **`_VIAL_HOLDER_REF_FIELD` / `_WELL_PLATE_HOLDER_REF_FIELD` constants
  removed**. Each was used exactly once. Inlined with a brief comment.
- **`load_deck_from_yaml_safe` docstring corrected** — `FileNotFoundError`
  is wrapped (kept for "user passed a bad path" ergonomics), other OS
  errors (`PermissionError`, `IsADirectoryError`, etc.) propagate raw.

## Docs

- `README.md`: removed stale `tip_holder` reference from the holder list
  (matching the Round-1 AGENTS.md cleanup that missed this line). Removed
  a newly-introduced rotting arithmetic comment (`z = 164.0 + 18.0 = 182.0`)

- `holder.py:131` bypass comment now states that `validate_assignment=True`

  it in isolation understands why the `object.__setattr__` is required.
- Misleading "Legacy dotted lookup still works via contained_labware mirror"

  labware name" semantics (previous key `"plate"` was literal; new key is

## Issues found during the round-2 fix work

- **`validate_assignment=True` does NOT roll back the field on validation
  failure.** When `holder.vials = bad_dict` raises, the field IS set, then
  the validator raises. The holder is now in an inconsistent state. Pinned
  this behavior in the new `test_vial_holder_validate_assignment_reruns_drift_check`
  test so it's explicit for future maintainers. The proper fix (route
  mutations through explicit `add_vial`/`remove_vial` methods and disable
  `validate_assignment=True`) is out of scope for this round.
- **The `seat_height is None` hard-error branches in both holder
  validators were dead code.** Both subclasses narrow
  `labware_seat_height_from_bottom_mm` to non-optional `float`, so the
  field-type validation rejects `None` before the model validator runs.
  Deleted the branches; the field type already enforces the invariant.
