# Progress — 2026-03-12

## Fix: Connect Hang When Mill is in Alarm State

### Problem
When the gantry was in alarm state, clicking Connect in the zoo UI caused it to hang indefinitely with no feedback to the user.

### Root Cause (two layers)
1. `Mill.connect_to_mill()` called `check_for_alarm_state()` which auto-reset alarms via `self.reset()` (sends `$X`) — can block on serial reads. Removed this call.

2. Even after removing auto-reset, `connect_to_mill()` still ran `read_mill_config()` (sends `$$`), `set_feed_rate()` (sends G-code), etc. **GRBL rejects all commands except `$X` and `$H` while in alarm state**, so these would hang or fail.

### Fix
1. **Early alarm detection in `Mill.connect_to_mill()`** — immediately after serial connect, sends `?` to check status. If alarm detected, skips all setup commands (`read_mill_config`, `set_feed_rate`, etc.) and returns immediately. The connection is established but no commands are sent that GRBL would reject.

2. **Removed `check_for_alarm_state()` call** — no more auto-reset during connect. The Gantry class's `_check_alarm_state()` detects alarm via `query_raw_status()` and logs a warning.

3. **30-second timeout on zoo's `/api/gantry/connect` endpoint** — connect runs in a background thread. If it doesn't complete in time, returns HTTP 504 with a clear message instead of hanging.

4. **Alarm state surfaces to UI** — after connect, position response's `status` field shows "Alarm", and the UI already displays the unlock button.

### Files Changed
- `PANDA_CORE/src/gantry/gantry_driver/driver.py` — added early alarm check in `connect_to_mill()` that skips setup commands when alarmed; removed `check_for_alarm_state()` call
- `zoo/zoo/routers/gantry.py` — added 30s timeout wrapper around `_gantry.connect()`, added `HTTPException` pass-through in error handling

### Tests
All 27 gantry tests pass.

## Feature: Instrument Connect Lifecycle + Mock/Dry Run Mode

### Problem
Protocol execution fails because instruments are instantiated from board YAML but never `connect()`ed. Real drivers (Filmetrics, UVVisCCS, Pipette) need active connections to operate. Additionally, there was no way to test a protocol without real hardware.

### Changes

#### 1. Board connect/disconnect lifecycle (`src/board/board.py`)
- `connect_instruments()` — iterates all instruments and calls `connect()` on each
- `disconnect_instruments()` — iterates all instruments and calls `disconnect()` on each, swallowing errors so one failing instrument doesn't block the rest

#### 2. Board loader mock_mode (`src/board/loader.py`)
- `load_board_from_yaml()` and `load_board_from_yaml_safe()` accept `mock_mode=False`
- When `mock_mode=True`, swaps each instrument type for its mock variant (e.g. `"pipette"` -> `"mock_pipette"`) if the mock exists in the registry
- Already-mock types and types without a mock variant are left unchanged

#### 3. Protocol setup threading (`src/protocol_engine/setup.py`)
- `setup_protocol()` and `run_protocol()` accept `mock_mode=False`
- `run_protocol()` now calls `board.connect_instruments()` before execution and `board.disconnect_instruments()` in a `finally` block

#### 4. Zoo dry_run endpoint (`zoo/zoo/routers/protocol.py`)
- `RunProtocolRequest` gains `dry_run: bool = False`
- When `dry_run=True`, uses `OfflineGantry` (no real gantry needed) and passes `mock_mode=True` to `run_protocol()`
- Skips gantry connected/homed checks for dry runs

#### 5. Zoo frontend (`zoo/frontend/`)
- API client accepts `dry_run?: boolean` in run request body
- ProtocolEditor has new "Dry Run" button (outlined green style) next to "Run Protocol"
- App.tsx wires `onDryRun` to call `handleRunProtocol(true)`

### Tests Added
- `tests/board/test_board.py` — 6 new tests: connect calls each instrument, propagates errors, disconnect calls each, continues after failure, empty board noop
- `tests/board/test_board_loader.py` — 5 new tests: mock_mode swaps types, leaves already-mock types, swaps all instruments, safe loader passes through
- `tests/setup/test_protocol_setup.py` — 4 new tests: mock_mode swaps types, run_protocol connects/disconnects, disconnects on failure, mock_mode run

#### 6. Remove mocks from Board tab UI (`zoo/zoo/routers/board.py`)
- Filtered out `mock_*` types from both `/instrument-types` and `/instrument-schemas` endpoints
- Users only see real instrument types (pipette, uvvis_ccs, filmetrics) in the Board editor dropdown
- Mock variants are still in the registry for programmatic use (mock_mode, tests)

### Verification
All 627 tests pass. Zoo frontend TypeScript compiles cleanly.
