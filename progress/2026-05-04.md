# 2026-05-04

## Agent architecture hardlines

Hydratone asked for clearer CubOS hard stops for coding agents so they preserve the intended abstraction boundaries instead of reaching into low-level gantry/G-code layers.

Work done:
- Added an `Architecture Hardlines for Agents` section to `AGENTS.md`.
- Documented the intended boundary stack: protocol intent -> `Deck` target resolution -> `Board` instrument offsets -> `Gantry` safe motion API -> `gantry_driver`/`Mill` serial and G-code.
- Added explicit hard stops against raw G-code outside the gantry driver, bypassing `Board.move()`, duplicating offset/labware geometry logic, frontend duplication of CubOS hardware logic, permissive config schemas, implicit mock fallback, and ad hoc persistence outside `DataStore`.

Notes:
- This is a docs-only change.
- Calibration script/operator UX hardlines were intentionally not included because Hydratone is actively working on calibration separately.

## Correction: closed-loop instrument exceptions

Hydratone corrected the first draft: the architecture hardlines cannot be absolute because some closed-loop instruments, such as ASMI indentation, legitimately coordinate gantry motion with sensor readings.

Updated `AGENTS.md` so low-level gantry access is a default prohibition with explicit exception rules. ASMI-style motion+sensing loops are allowed when documented, tested with focused mocks, and hidden behind protocol/business abstractions so end users still do not write G-code or know GRBL details.
