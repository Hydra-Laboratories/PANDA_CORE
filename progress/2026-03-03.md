# 2026-03-03 — Hardware Test Scripts for Gantry Axis Refactor

## Work Done

Created 4 hardware test scripts for the `gantry-axis-refactor` PR to validate the positive-space coordinate system on the Genmitsu 3018 PRO Desktop CNC.

### Scripts Created

1. **`tests/hardware/test_boundary_moves.py`** — Moves to every edge and corner of the working volume (14 test points: 4 XY corners at z_min, 4 at z_max, 6 axis edge midpoints). Returns to center after each move and verifies accuracy.

2. **`tests/hardware/test_custom_position.py`** — Moves to a user-specified X/Y/Z coordinate. Supports `--repeat N` for consistency checking, `--force` for out-of-bounds override, and bounds validation before sending commands.

3. **`tests/hardware/test_out_of_bounds.py`** — Software-only test (no hardware needed). Validates that `WorkingVolume.contains()` correctly rejects 6 out-of-bounds coordinates and accepts 5 in-bounds coordinates. All 11 checks pass.

4. **`tests/hardware/test_coordinate_translation.py`** — Verifies coordinate translation between user-space (positive) and machine-space (negative) at 9 test points. Checks raw status, translated status, and `get_coordinates()` output.

### Common Features Across All Scripts

- `--dry-run` flag prints expected movements and translations without hardware
- `--config` flag defaults to Desktop CNC config
- `--i-understand-risk` safety flag required for live runs
- Prints both user-space and machine-space coordinates for every move
- Reuses `ResultTracker`, `build_safe_xy_corners()`, `working_volume_from_config()`, `is_within_tolerance()` from existing code

### Verification

All 4 scripts verified in dry-run mode:
- `test_boundary_moves.py --dry-run` — 14 targets displayed correctly
- `test_custom_position.py --x 150 --y 100 --z 10 --dry-run` — translation shown
- `test_out_of_bounds.py` — 11/11 bounds checks passed
- `test_coordinate_translation.py --dry-run` — 9 test points displayed

Safety guards verified:
- Scripts refuse to run without `--i-understand-risk`
- Custom position test correctly identifies out-of-bounds targets

### Issue Found & Fixed

When running scripts sequentially, the machine retains its WPos/MPos across serial reconnections — GRBL doesn't reset position on disconnect. The first script left the machine at a non-origin position, so the second script started from there and could exceed bounds.

**Fix applied to all 3 hardware scripts:**
1. **Print current position on connect** — shows WPos/MPos immediately after connecting so you know the starting state
2. **Return to origin (0,0,0) before disconnecting** — leaves the machine in a known state for the next script

### Key Observation Documented

`Gantry.move_to()` does NOT perform bounds checking against `WorkingVolume` before sending commands. The `test_out_of_bounds.py` script documents this gap and the note is printed after its run.
