# 2026-04-22 — ASMI protocol runner + scan motion clarification

## Context

ASMI protocol bring-up uncovered three separate sources of confusion:

1. `setup/run_protocol.py` connected the gantry but did not connect board
   instruments before executing the protocol, so ASMI could reach
   `indentation()` with the Vernier sensor still disconnected.
2. The gantry often powers up in GRBL alarm state, which is expected on this
   machine. The runner needed to clear that state safely before protocol
   execution.
3. Scan transit semantics were ambiguous:
   - `safe_z` is a protocol named position, not a deck concept.
   - `move(..., travel_z=...)` is for raw/named XYZ transits.
   - `scan.entry_travel_z` is only for the first transit into the scan.
   - `scan.safe_approach_height` is only for within-scan well-to-well travel.
   - Board-level `safe_approach_height` still means a **relative** offset for
     generic `move_to_labware()`-style motion; scan-level
     `safe_approach_height` is now an **absolute Z** override.

## Work done

### Protocol runner / startup

- `setup/run_protocol.py`
  - Now clears the expected gantry alarm state before execution.
  - Now connects all board instruments before `protocol.run(context)`.
  - Now disconnects instruments in `finally`, alongside the gantry cleanup.

- `src/gantry/gantry.py`
  - Added `prepare_for_protocol_run()` to:
    - query raw GRBL status
    - unlock if the controller is in alarm
    - restore controller state that `connect_to_mill()` intentionally skips
      when connecting into an alarmed GRBL controller

### Protocol motion semantics

- `src/protocol_engine/commands/move.py`
  - Added optional `travel_z` for named/literal XYZ moves.
  - This guarantees the desired `Z -> XY -> final Z` ordering for raw/named
    positions such as protocol `safe_z`.
  - Deck targets explicitly reject `travel_z`; deck-target motion still routes
    through `move_to_labware()`.

- `src/protocol_engine/commands/scan.py`
  - Added `entry_travel_z` as an absolute Z used only for the first move into
    the scan (e.g. the initial transit to A1).
  - `safe_approach_height` on `scan` is now documented and implemented as an
    absolute Z used only for within-scan well-to-well travel.
  - Final retract after the scan uses the scan's final approach Z.

- `src/protocol_engine/commands/_movement.py`
  - Protocol-level scan overrides bypass `Board.move_to_labware()` and emit an
    explicit absolute-Z transit move with `travel_z=approach_z`, followed by
    the descent to the action Z.
  - Added validation that an absolute scan approach Z cannot sit below the
    action Z.

### ASMI config

- `configs/protocol/asmi_indentation.yaml`
  - Added `entry_travel_z: 20.0`
  - Set `safe_approach_height: 20.0`
  - Kept `safe_z` as a protocol named position used by the later `move` step,
    not as a deck/labware field
  - `move: safe_z` now also uses `travel_z: 20.0`

## Important current semantics

- The repo still uses the existing high-level positive-down Z convention.
- ASMI has two different `measurement_height` meanings:
  - Board YAML `measurement_height` = relative instrument offset used by
    generic movement helpers.
  - `ASMI.indentation(..., measurement_height=...)` = absolute Z start point
    for the indentation itself.
- Only the `scan` command special-cases `safe_approach_height` as an absolute
  Z override. Board/instrument `safe_approach_height` remains a relative
  offset used by generic `move_to_labware()` behavior.
