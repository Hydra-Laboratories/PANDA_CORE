## Design Decisions

- Holder `location` is currently treated as a generic deck-space reference point / anchor.
- Bounding-box dimensions are stored now for future collision modeling, but collision math/orientation is not implemented in this task.
- Holder `slots` remain available as metadata-only addressable coordinates, but holders can now also instantiate nested `Vial`/`WellPlate` children for experiment target generation.
- Holder slot Z defaults to the holder's resolved Z when omitted in YAML.
- `height_mm` continues to represent the overall holder collision envelope. For `WellPlateHolder`, the newly encoded `10.0 / 5.0` values are stored separately as seat/support metadata so the earlier `14.8` mm envelope is preserved.

## Open Questions

- What physical reference should each holder `location` represent long-term: center, front-left corner, A1-equivalent, or another fixture datum?
- For `WellPlateHolder`, should the holder expose a single canonical slot name like `plate`, or should it eventually resolve directly into the contained well plate geometry?
- For `VialHolder`, what are the exact nine vial slot coordinates and naming convention (`vial_1`..`vial_9`, row/column labels, etc.)?
- Should holders eventually own/contain nested labware instances, or should deck YAML keep holders and contained labware as separate top-level objects linked by slot names?

## Design Follow-Up

- Revisited the current A1/orientation approach:
  - The existing `WellPlate` two-point calibration is good for manually measured placement because it captures both anchor and orientation from real deck points.
  - It is less ideal as the universal representation for all labware, especially holders, because fixtures do not naturally have an `A1` semantic and the current generic `location` field is ambiguous.
- Recommended next-step design direction:
  - Separate `anchor` semantics from `orientation`.
  - Use explicit anchor metadata such as `anchor_type` (`a1_center`, `slot_1_center`, `center`, `drop_center`, etc.).
  - Use a simple UI/YAML-friendly orientation field such as `rotation_deg: 0|90|180|270` for CAD-defined fixtures and generated layouts.
  - Keep the existing two-point calibration mode for physically measured well plates.
- New user requirement:
  - Holder seat height should participate directly in experiment Z generation.
  - This implies the deck YAML model should support defining a holder and then defining contained labware positions relative to that holder, rather than treating holder slots as passive metadata only.
  - The same nested-holder model should apply to both vial holders and well plate holders.

## Panda YAML Assumptions

- `configs/deck/panda_deck.yaml` preserves the exact tip pickup and XY vial-center coordinates from `panda.json`.
- The well plate orientation was inferred as:
  - `orientation = 0` -> columns increase along `+X`
  - rows increase along `+Y`
  - therefore `A2 = A1 + (9.0, 0.0, 0.0)` and `B1 = A1 + (0.0, 9.0, 0.0)`
- The vial holder bottom Z was inferred as `182.0 - 18.0 = 164.0`.
- The well plate holder bottom Z was inferred as `188.0 - 5.0 = 183.0`.
- The panda deck now uses nested holder children, so contained vial / well-plate experiment Z is generated from holder seat height instead of being hard-coded in child YAML.
- `panda.json` includes `vol_ul` per vial (current fill state), but the deck schema currently models `working_volume_ul` rather than current fill. The YAML fixture therefore preserves geometry and capacity while using `working_volume_ul = capacity_ul` instead of incorrectly mapping `vol_ul` into configuration geometry.
