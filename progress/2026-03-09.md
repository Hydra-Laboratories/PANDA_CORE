**`data/data_reader.py`** (new) ‚Äî Read-only query interface

- `CampaignRecord`, `ExperimentRecord`, `LabwareRecord` frozen dataclasses
- `get_campaign()`, `list_campaigns()` ‚Äî campaign metadata + config snapshots
- `get_experiments()` ‚Äî filter by campaign, labware_name, well_id
- `get_labware()` ‚Äî filter by campaign, labware_key
- `get_measurements()` ‚Äî generic raw rows from any measurement table (with SQL injection guard)
- `get_measurements_by_campaign()` ‚Äî joins experiments to get well_id/labware_name with measurements
- Context manager support

**`data/analysis/uvvis.py`** (new) ‚Äî UV-Vis specific read + analysis
- `UVVisRecord` frozen dataclass (unpacked wavelengths, intensities, metadata)
- `unpack_spectrum()` ‚Äî BLOB ‚Üí tuple[float, ...] deserialization
- `load_uvvis_by_experiment()` / `load_uvvis_by_campaign()` ‚Äî load spectra from DB via DataReader
- `peak_wavelength()` ‚Äî find (wavelength, intensity) at max intensity
- `absorbance()` ‚Äî compute A = -log10((sample - dark) / (reference - dark))
- `slice_wavelength_range()` ‚Äî filter spectrum to [wl_min, wl_max]

### Design
- **DataReader** is generic (campaigns, experiments, labware, any measurement table)
- **analysis/uvvis.py** is instrument-specific (BLOB format, spectral operations)
- DataReader accepts a `connection` param for sharing with DataStore in tests
- Valid measurement tables are whitelisted to prevent SQL injection

## Coordinate Convention Update: WPos (Positive X/Y)

### Background
PANDA_CORE uses GRBL WPos (work position) reporting (`$10=0`). After `xy_hard_limits` homing, position is (0,0,0) and X/Y travel is into **positive** space. Z is still negative (down from home).

The old mofcat-workflow used MPos (machine position) coordinates ‚Äî a different convention:
- `X_HOME=-157`, `Y_HOME=67` (MPos)
- `A1: x=-37.4, y=38.134, z=8.116` (MPos)
- These values **cannot be used directly** in PANDA_CORE; they need recalibration in WPos.

### Changes
- Updated `WorkingVolume` docstring in `src/gantry/gantry_config.py` ‚Äî removed "extends into negative space"
- Updated gantry configs to use positive X/Y bounds:
  - `configs/gantry/genmitsu_3018_PRO_Desktop.yaml`: x 0‚Üí300, y 0‚Üí200, z -80‚Üí0
  - `configs/gantry/genmitsu_3018_PROver_v2.yaml`: x 0‚Üí300, y 0‚Üí200, z -80‚Üí0
- No code/schema changes needed ‚Äî validation only enforces `min < max`, no sign constraints

### Key Convention
- **X/Y: positive** (0 to max) ‚Äî WPos after homing
- **Z: negative** (0 at top, negative going down) ‚Äî TBD, may also be positive
- Mofcat-workflow MPos values need converting; deck calibration positions must be re-measured in WPos on the physical machine
